How I Structure a Production Django Project in 2026

A real-world layout for apps, services, settings, and tests — battle-tested across 12 production deployments, four time zones, and three SaaS rewrites.

SH
Shahriar HosenSenior Full Stack Developer
May 24, 2026 7 min read 12.4k views 24 comments

Most Django tutorials stop at startproject. After three years of shipping production SaaS, I've landed on a layout that survives growth: clear domain boundaries, predictable settings, a service layer, and tests that run in under a minute.

1. The folder layout

Keep the project shallow. Two top-level directories — apps/ and config/ — and a single manage.py. Resist the urge to invent four layers of nesting.

project/
├── apps/
│   ├── accounts/
│   ├── billing/
│   ├── catalog/
│   └── core/
├── config/
│   ├── settings/
│   │   ├── base.py
│   │   ├── dev.py
│   │   └── prod.py
│   ├── urls.py
│   └── wsgi.py
├── tests/
└── manage.py

2. A service layer that earns its keep

Views shouldn't talk to the ORM directly. Put business logic in services.py inside each app. Your serializers become thin, your tests become fast, and your views become boring — which is exactly what you want.

"If a junior dev can't find where an action happens in 30 seconds, your structure is wrong." — me, after one too many code reviews.

3. Settings without surprises

Split settings.py into base, dev, and prod. Read secrets from environment variables via django-environ. Never commit a .env. Validate required variables on boot — failing fast at startup beats failing slow at 3am.

4. Testing strategy

  • Unit tests for services. Fast, no database where possible.
  • API tests for endpoints. Use DRF's APITestCase.
  • Factory Boy for fixtures — no JSON dumps to maintain.
  • Run with pytest -n auto for parallel runs.

5. Deployment-ready from day one

Dockerise on day one. Add a Makefile with make up, make test, make migrate. New teammates should be productive in 15 minutes.

Wrap-up

This isn't theory — it's the same layout running behind production SaaS handling millions of requests a month. Steal it, adapt it, but commit to one structure and let the team trust it.
